## Проблема

Малому бизнесу, фрилансерам и начинающим предпринимателям часто нужен сайт, но создание сайта занимает много времени и требует разных навыков:

- нужно придумать структуру сайта;
- написать продающие тексты;
- подобрать визуальный стиль;
- сверстать страницу;
- разместить сайт в интернете.

## Решение

Наш сервис позволяет создать сайт по текстовому описанию.

Пользователь вводит, например:

Сайт для студии йоги в Казани. Нужно показать направления занятий, тренеров, расписание и дать возможность записаться на пробное занятие.

После этого ИИ предлагает структуру сайта:
 - Главная;
 - Направления;
 - Тренеры;
 - Расписание;
 - Контакты.

Затем система генерирует страницы, тексты и базовый дизайн.

## Ценность продукта

Продукт позволяет пользователю получить готовый сайт за несколько минут, даже если он не умеет программировать, писать тексты и делать дизайн.

Главная ценность:
- быстрое создание сайта;
- простой интерфейс;
- генерация структуры и текстов с помощью ИИ;
- возможность редактирования результата;
- публикация готовой страницы по ссылке.

## Поддерживаемые типы сайтов

В MVP поддерживаются:
 1. Лендинг
Одностраничный сайт для заявки, записи, презентации услуги или мероприятия.
 2. Многостраничный сайт
Простой сайт до 5 страниц для компании, специалиста, студии, школы, портфолио или локального бизнеса.

**Система может генерировать такие страницы:**
 - Главная;
 - О нас;
 - Услуги;
 - Портфолио;
 - Команда;
 - FAQ;
 - Контакты;
 - Меню / Каталог;
 - Расписание.

## Основной пользовательский сценарий

1. Пользователь открывает сервис.
2. Вводит описание бизнеса или проекта.
3. Выбирает тип сайта.
4. Выбирает цель сайта.
5. Нажимает кнопку “Сгенерировать”.
6. Система создаёт структуру сайта.
7. Пользователь видит предварительный результат.
8. Пользователь может отредактировать тексты.
9. Пользователь публикует сайт.
10. Система выдаёт публичную ссылку.

## Функциональные требования

Требования:
- на странице должно быть поле для ввода описания;
- пользователь может описать бизнес, услугу или проект;
- если поле пустое, система должна показать ошибку;
- описание используется для генерации структуры сайта.

 1. Выбор типа сайта

Пользователь должен выбрать, какой сайт он хочет создать.

Варианты:

- одностраничный сайт;
- многостраничный сайт.

Требования:

- пользователь выбирает один вариант;
- выбранный тип влияет на структуру сайта;
- для многостраничного сайта система создаёт несколько страниц.

2. Выбор цели сайта

Пользователь должен выбрать главную цель сайта.

Варианты:

- получить заявки;
- записать клиента;
- рассказать о компании;
- показать услуги;
- показать портфолио;
- рассказать о мероприятии.

Требования:

- пользователь выбирает одну цель;
- выбранная цель учитывается при генерации сайта;
- если цель не выбрана, система должна показать ошибку.

3. Выбор стиля сайта

Пользователь должен выбрать стиль будущего сайта.

Требования:

- пользователь описывает проект, цель, оформление и нужную информацию.
- ИИ самостоятельно определяет структуру, стиль, блоки и тексты сайта.

4. Генерация сайта

Система должна создать сайт на основе описания пользователя и карты сайта.

Требования:

- система генерирует страницы сайта;
- система генерирует заголовки;
- система генерирует тексты;
- система создаёт базовую структуру блоков;
- главная страница должна содержать первый экран;
- контактная страница должна содержать контактный блок.

5. Предпросмотр сайта

Пользователь должен увидеть сгенерированный сайт до публикации.

Требования:

- пользователь видит готовую страницу сайта;
- пользователь может переключаться между страницами;
- сайт отображается понятно и без ошибок;
- на сайте есть меню для перехода между страницами.

6. Редактирование 

Пользователь должен иметь возможность изменить основные тексты сайта.

В MVP пользователь может изменять результат сайта через уточнение запроса и повторную генерацию:

— описание проекта;
— название компании;
— тип сайта;
— главную цель сайта;
— пожелания к оформлению;
— информацию, которую нужно разместить на сайте;
— почту и номер телефона;
— количество кнопок;
— уточнение, если первый вариант сайта не понравился.

Требования:

— пользователь может ввести или изменить данные перед генерацией сайта;
— пользователь может посмотреть сгенерированный сайт в предпросмотре;
— пользователь может написать уточнение и перегенерировать сайт;
— изменения учитываются ИИ при повторной генерации;
— после генерации пользователь может получить публичную ссылку на сайт.

7. Сохранение проекта

Система должна сохранять созданный сайт.

Требования:

- сохраняется структура сайта;
- сохраняются страницы;
- сохраняются тексты;
- сохраняются изменения пользователя.

8. Публикация сайта

Пользователь должен иметь возможность опубликовать сайт.

Требования:

- на странице есть кнопка “Опубликовать”;
- после нажатия система создаёт публичную ссылку;
- пользователь видит эту ссылку;
- сайт открывается по ссылке без входа в аккаунт.

9. Просмотр опубликованного сайта

Опубликованный сайт должен открываться по публичной ссылке.

Требования:

- ссылка открывается в браузере;
- сайт отображается без ошибок;
- пользователь может переходить между страницами;
- отображается последняя сохранённая версия сайта.

## Нефункциональные требования

1. Простота интерфейса

Интерфейс должен быть понятен человеку без технических знаний.

Пользователь должен понимать:

- куда ввести описание;
- что выбрать;
- какую кнопку нажать;
- где посмотреть результат;
- как опубликовать сайт.

2. Скорость работы

Система не должна “зависать” без объяснения.

Если генерация занимает время, нужно показывать экран загрузки или текст:

“Сайт генерируется, пожалуйста, подождите.”

3. Понятные ошибки

Если что-то пошло не так, пользователь должен увидеть понятное сообщение.

Примеры:

- “Введите описание сайта.”
- “Выберите цель сайта.”
- “Не удалось сгенерировать сайт. Попробуйте ещё раз.”

4. Адаптивность

Интерфейс должен нормально отображаться на ноутбуке.

Желательно, чтобы сайт также корректно отображался на телефоне, но для первого MVP главное — рабочая версия для ноутбука.

5. Надёжность

Система должна сохранять данные пользователя.

Если пользователь изменил текст и нажал “Сохранить”, изменения не должны пропадать.

6. Расширяемость

Проект должен быть сделан так, чтобы в будущем можно было добавить:

- новые типы страниц;
- новые блоки;
- новые стили;
- генерацию изображений;
- экспорт сайта;
- личный кабинет.

## Риски

**Риск 1. Слишком большой объём проекта**

Команда может не успеть сделать полноценный конструктор сайтов.

Как снизить риск:

- ограничить сайт максимум 5 страницами;
- не делать интернет-магазин;
- не делать сложный редактор;

**Риск 2. ИИ может сгенерировать плохой результат**

ИИ может вернуть странные тексты, неправильную структуру или неполные данные.

Как снизить риск:

- заранее задать список разрешённых страниц;
- заранее задать список разрешённых блоков;
- использовать готовый формат JSON;
- проверять результат перед показом пользователю.

**Риск 3. Команда может не успеть сделать редактирование**

Редактор может оказаться сложнее, чем кажется.

Как снизить риск:

- не делать drag-and-drop;
- не делать сложные настройки дизайна.

**Риск 4. Сложность публикации сайта**

Публикация многостраничного сайта может занять время.

Как снизить риск:

- сделать простую публикацию по ссылке;
- использовать один публичный адрес для проекта;
- страницы открывать через простые маршруты.

**Риск 5. Непонятный интерфейс**

Пользователь может не понять, что делать дальше.

Как снизить риск:

- сделать пошаговый процесс;
- использовать понятные кнопки;
- показывать подсказки;
- не перегружать экран.

**Риск 6. Проблемы с распределением задач**

Участники команды могут делать разные вещи без общего понимания.

Как снизить риск:

- хранить требования в GitHub;
- фиксировать задачи в backlog;
- регулярно сверяться с критериями готовности;
- договориться, кто за что отвечает.
